This page provides detailed documentation for all available op-reth configuration options, organized by functionality.

The options below are exposed by `op-reth v2.2.3`.

For an upstream cross-reference see the [Reth CLI documentation](https://reth.rs/cli/reth.html).

<Info>
  **op-reth is the recommended execution client for new OP Stack deployments.** When pairing with op-node, remember to set `--l2.enginekind=reth` on op-node (op-node still defaults to `geth` for backward compatibility). See the [op-node configuration options](/node-operators/reference/op-node-config) page for the matching engine-side flags.
</Info>

## General Options

Options for configuring the node instance and chain.

### config

The path to the configuration file to use.

<Tabs>
  <Tab title="Syntax">`--config <FILE>`</Tab>
</Tabs>

### chain

The chain this node is running. Possible values are either a built-in chain or the path to a chain specification file.

Built-in chains:
```
optimism, optimism_sepolia, ink, ink-sepolia, soneium, soneium-minato-sepolia, unichain, unichain-sepolia, worldchain, worldchain-sepolia, etc
```

<Tabs>
  <Tab title="Syntax">`--chain <CHAIN_OR_PATH>`</Tab>
  <Tab title="Default">`optimism`</Tab>
</Tabs>

### instance

Add a new instance of a node.

Configures the ports of the node to avoid conflicts with the defaults. This is useful for running multiple nodes on the same machine.

Max number of instances is 200. It is chosen in a way so that it's not possible to have port numbers that conflict with each other.

Changes to the following port numbers:

- `DISCOVERY_PORT`: default + `instance` − 1
- `AUTH_PORT`: default + (`instance` × 100) − 100
- `HTTP_RPC_PORT`: default − `instance` + 1
- `WS_RPC_PORT`: default + (`instance` × 2) − 2
- `IPC_PATH`: default + `-instance`

<Tabs>
  <Tab title="Syntax">`--instance <INSTANCE>`</Tab>
</Tabs>

### with-unused-ports

Sets all ports to unused, allowing the OS to choose random unused ports when sockets are bound.

Mutually exclusive with `--instance`.

<Tabs>
  <Tab title="Syntax">`--with-unused-ports`</Tab>
</Tabs>

## Datadir

Options for data storage.

### datadir

The path to the data dir for all reth files and subdirectories.

Defaults to the OS-specific data directory:

- Linux: `$XDG_DATA_HOME/reth/` or `$HOME/.local/share/reth/`
- Windows: `{FOLDERID_RoamingAppData}/reth/`
- macOS: `$HOME/Library/Application Support/reth/`

<Tabs>
  <Tab title="Syntax">`--datadir <DATA_DIR>`</Tab>
  <Tab title="Default">`default`</Tab>
</Tabs>

### datadir.static-files

The absolute path to store static files in.

<Tabs>

</Tabs>

### datadir.rocksdb

The absolute path to store `RocksDB` database in.

<Tabs>
  <Tab title="Syntax">`--datadir.rocksdb <PATH>`</Tab>

### datadir.pprof-dumps

Experimental, for usage in research. Sets the max accumulated byte size of transactions to pack in one response. Spec'd at 2MiB.

  <Tab title="Syntax">`--pooled-tx-response-soft-limit <BYTES>`</Tab>
  <Tab title="Default">`2097152`</Tab>
</Tabs>

### pooled-tx-pack-soft-limit

Experimental, for usage in research. Sets the max accumulated byte size of transactions to request in one request.

Since `RLPx` protocol version 68, the byte size of a transaction is shared as metadata in a transaction announcement (see `RLPx` specs). This allows a node to request a specific size response.

By default, nodes request only 128 KiB worth of transactions, but should a peer request more, up to 2 MiB, a node will answer with more than 128 KiB.

Default is 128 KiB.

<Tabs>
  <Tab title="Syntax">`--pooled-tx-pack-soft-limit <BYTES>`</Tab>
  <Tab title="Default">`131072`</Tab>
</Tabs>

### max-tx-pending-fetch

<Tabs>
  <Tab title="Syntax">`--max-tx-pending-fetch <COUNT>`</Tab>
  <Tab title="Default">`25600`</Tab>
</Tabs>

### tx-channel-memory-limit

Memory limit (in bytes) for the channel that buffers transaction events flowing from the network manager to the transactions manager.

When the budget is exhausted, new events are dropped (see metric `total_dropped_tx_events_at_full_capacity`). Acts as a backstop against unbounded memory growth under sustained P2P transaction flooding.

<Tabs>
  <Tab title="Syntax">`--tx-channel-memory-limit <BYTES>`</Tab>
  <Tab title="Default">`1073741824`</Tab>
</Tabs>

### net-if.experimental

<Tabs>

<Tabs>
  <Tab title="Syntax">`--disable-tx-gossip`</Tab>
</Tabs>

Examples: sqrt, all, max:10

<Tabs>

  <Tab title="Default">`sqrt`</Tab>
</Tabs>

### required-block-hashes

Comma separated list of required block hashes or block number=hash pairs. Peers that don't have these blocks will be filtered out. Format: hash or `block_number=hash` (e.g., 23115201=0x1234...)

<Tabs>
  <Tab title="Syntax">`--required-block-hashes <REQUIRED_BLOCK_HASHES>`</Tab>
</Tabs>

### network-id

Optional network ID to override the chain specification's network ID for P2P connections

<Tabs>
  <Tab title="Syntax">`--network-id <NETWORK_ID>`</Tab>
</Tabs>

### eth-max-message-size

Maximum allowed ETH message size in bytes. Default is 10 MiB

<Tabs>
  <Tab title="Syntax">`--eth-max-message-size <BYTES>`</Tab>
</Tabs>

### netrestrict

Restrict network communication to the given IP networks (CIDR masks).

Comma separated list of CIDR network specifications. Only peers with IP addresses within these ranges will be allowed to connect.

Example: --netrestrict "192.168.0.0/16,10.0.0.0/8"

<Tabs>
  <Tab title="Syntax">`--netrestrict <NETRESTRICT>`</Tab>
</Tabs>

### enforce-enr-fork-id

Enforce EIP-868 ENR fork ID validation for discovered peers.

When enabled, peers discovered without a confirmed fork ID are not added to the peer set until their fork ID is verified via EIP-868 ENR request. This filters out peers from other networks that pollute the discovery table.

<Tabs>
  <Tab title="Syntax">`--enforce-enr-fork-id`</Tab>
</Tabs>

### engine.share-sparse-trie-with-payload-builder
